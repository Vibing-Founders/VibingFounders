# Authentication & Session Management Patterns

> Reference guide to secure authentication, session management, token handling, MFA, and OAuth patterns for web and mobile platforms.

---

## 1. Password Authentication

### Password storage

**Never store passwords in plaintext or with reversible encryption.** Always use a one-way cryptographic hash with a salt.

**Recommended algorithms (in order of preference):**
1. **Argon2id** — winner of the Password Hashing Competition; memory-hard, resistant to GPU/ASIC attacks
2. **bcrypt** — widely deployed, well-understood; use cost factor 10–12 minimum (increase as hardware improves)
3. **scrypt** — memory-hard; good choice but more complex to tune
4. **PBKDF2** — NIST-approved; acceptable but less strong than the above against GPU attacks

**Never use** for passwords: MD5, SHA-1, SHA-256/512 without salting, any un-salted hash, any fast hash.

**Implementation notes:**
- Use a library, don't implement hashing yourself
- The library handles salting automatically — don't add your own salt on top
- Store the full output string (algorithm identifier + salt + hash), not just the hash
- When verifying: use constant-time comparison to prevent timing attacks

### Password policies

**Minimum requirements:**
- Minimum length: 8 characters (NIST now recommends up to 64 characters allowed)
- Reject passwords found in known breach lists (HaveIBeenPwned API — k-anonymity model so you never send the full hash)
- Allow all printable ASCII characters and unicode
- Do NOT require forced rotation schedules (NIST guidance: only rotate on suspected compromise)
- Do NOT require complexity rules (uppercase + number + symbol) — they produce weak, predictable patterns

### Password reset

- Use a cryptographically random, single-use token (min 128 bits entropy)
- Tokens must expire quickly: **1 hour maximum**, preferably 15–30 minutes
- Send reset link via email only (do not expose token in URLs that could be logged)
- Invalidate all other sessions on successful password reset
- Notify the user that a password reset occurred via their registered email
- Rate limit reset requests per account and per IP to prevent email bombing

## 2. Multi-Factor Authentication (MFA)

### MFA factors

| Factor type | Examples | Notes |
|---|---|---|
| Something you know | Password, PIN | First factor only; not a second factor |
| Something you have | TOTP app (Authenticator), hardware key, SMS OTP | TOTP app or hardware key preferred |
| Something you are | Biometrics (fingerprint, face) | Usually device-local, not sent to server |

### Recommended MFA approaches

**TOTP (Time-based One-Time Password)** — RFC 6238
- Uses a shared secret + current timestamp to generate a 6-digit code that changes every 30 seconds
- Apps: Google Authenticator, Authy, 1Password, any TOTP-compatible app
- Server validates codes by checking current window and ±1 window (to allow for clock drift)
- Store the shared secret encrypted at rest; show it once at setup (as QR code) then never again
- Provide backup codes (10× 8-character random codes) at setup for account recovery

**Hardware security keys** (FIDO2/WebAuthn)
- Strongest MFA option — phishing-resistant because the key signs a challenge specific to the origin
- Supported natively in modern browsers
- Best for high-value accounts (admins, creators with payout access)

**SMS OTP** — acceptable but weakest option
- Vulnerable to SIM swapping, SS7 attacks, and phishing
- Acceptable as a fallback, not as the primary MFA option
- Never use SMS as the only MFA option for high-value accounts

**MFA enforcement:**
- Require MFA for: all admin accounts, all accounts with access to financial data, all accounts with moderation/elevated privileges
- Strongly encourage (but don't require) MFA for all users
- After successful MFA, mark the session as MFA-verified; require re-auth for sensitive operations

## 3. Session Management

### Session token properties

A session token must be:
- **Random**: generated by a CSPRNG (cryptographically secure pseudorandom number generator)
- **Unpredictable**: minimum 128 bits of entropy (typically a 32-character hex string or 43-character base64url)
- **Unique**: never reused
- **Not derived from user data**: never use `md5(user_id + timestamp)` or similar

### Session token transmission

**Cookies (recommended for web apps):**
- Set `HttpOnly` — prevents JavaScript access, mitigates XSS theft
- Set `Secure` — only transmitted over HTTPS
- Set `SameSite=Strict` or `SameSite=Lax` — mitigates CSRF attacks
  - `Strict`: cookie never sent on cross-site requests (most secure; can break some flows)
  - `Lax`: cookie sent on top-level navigation but not on cross-origin resource requests (good default)
- Set appropriate `Max-Age` or `Expires`
- Example: `Set-Cookie: session=<token>; HttpOnly; Secure; SameSite=Lax; Max-Age=3600; Path=/`

**Authorization header (for APIs/mobile):**
- `Authorization: Bearer <token>`
- Store token in memory (not localStorage — vulnerable to XSS) or in a `HttpOnly` cookie
- Never store sensitive tokens in localStorage or sessionStorage if you have XSS risk

### Session lifecycle

- **Create new session** on login (do not reuse pre-auth session IDs — session fixation)
- **Invalidate session** on logout (server-side, not just client-side cookie deletion)
- **Invalidate all sessions** on password change
- **Rotate session token** after privilege escalation (e.g. after MFA completion)
- **Set inactivity timeout**: invalidate sessions idle for >30 minutes (shorter for sensitive operations)
- **Set absolute timeout**: force re-authentication after a maximum session lifetime (e.g. 24–48 hours)

### Concurrent sessions

Decide and enforce your policy:
- Allow multiple concurrent sessions (common for most platforms) — log each session separately
- Alert user on new device login
- Provide "log out all devices" functionality
- For high-risk operations (payouts, admin actions): consider requiring re-authentication even in an active session

## 4. JWT (JSON Web Tokens)

JWTs are commonly used for stateless auth but have several critical pitfalls.

### JWT structure

A JWT has three parts: `header.payload.signature`

**Header** — specifies algorithm:
```json
{"alg": "HS256", "typ": "JWT"}
```

**Payload** — claims:
```json
{"sub": "user_123", "role": "user", "iat": 1700000000, "exp": 1700003600}
```

**Signature** — HMAC or RSA/ECDSA signature of header + payload

### Critical security requirements

**1. Verify the signature** — always. Never trust a JWT that fails signature verification.

**2. Specify and validate the algorithm** — never accept `alg: none`. Allowlist acceptable algorithms. Prefer RS256/ES256 (asymmetric) over HS256 (symmetric) for systems with multiple consumers.

**3. Verify expiry (`exp`)** — always check and reject expired tokens.

**4. Set a short expiry** — access tokens: 15 minutes to 1 hour. Refresh tokens: days to weeks.

**5. Validate `iss` (issuer) and `aud` (audience)** if using multiple services.

**6. Include meaningful claims** — include the minimum needed. Don't put sensitive data in the payload (it's base64-encoded, not encrypted).

### Access token + refresh token pattern

```
Access token:  short-lived (15–60 min), used for API calls
Refresh token: long-lived (7–30 days), stored securely, used only to get a new access token
```

**Refresh token rotation**: issue a new refresh token every time you use one, and invalidate the old one. This allows detection of refresh token theft (if the old token is used, it means compromise → invalidate all refresh tokens for that user).

**Token storage on client:**
- Web: access token in memory; refresh token in `HttpOnly Secure` cookie
- Mobile: access token in memory; refresh token in secure storage (iOS Keychain, Android Keystore)
- Never store tokens in localStorage if XSS is possible

## 5. OAuth 2.0 / OpenID Connect

### Core flows

**Authorization Code Flow** (recommended for web and mobile):
1. App redirects user to auth server with `response_type=code`, `scope`, `state`, `code_challenge` (PKCE)
2. User authenticates and consents
3. Auth server redirects back with `code` and `state`
4. App validates `state`, exchanges `code` for tokens using PKCE `code_verifier`
5. Tokens returned: `access_token`, `refresh_token`, `id_token` (OIDC)

**PKCE (Proof Key for Code Exchange)**: Required for all public clients (mobile, SPAs). Prevents authorization code interception attacks. Always use PKCE even for confidential clients (it doesn't hurt and adds protection).

**Implicit Flow** (deprecated): Do not use. Returns tokens directly in the URL fragment — tokens can be logged, leaked via Referer headers, etc.

**Client Credentials Flow**: For machine-to-machine auth (backend services). No user involved. Use with short-lived tokens and rotate client secrets regularly.

### Critical security requirements

- **Validate `state` parameter**: prevents CSRF on the callback
- **Validate `redirect_uri`**: must be an exact match to registered URIs (no wildcards)
- **Use PKCE** for all public clients
- **Validate `id_token` signature and claims** (OIDC): `iss`, `aud`, `exp`, `nonce`
- **Protect client secrets**: never expose confidential client secrets in frontend code
- **Scope principle of least privilege**: request only the scopes actually needed

### Choosing OAuth scopes for your own API

Define scopes that match your trust tiers:
- `read:profile` — view own profile
- `write:profile` — update own profile
- `read:bookings` — view own bookings
- `admin:users` — admin user management
- `admin:moderation` — moderator actions

## 6. Identity Verification for High-Risk Features

For platforms where adults interact with minors, or where payouts are involved, basic username/password is not sufficient. You need **identity verification**.

### Verification levels and use cases

| Level | Methods | Use cases |
|---|---|---|
| L1 | Email + phone OTP | Basic account confirmation |
| L2 | Selfie + age estimation | Age-banding for teen features |
| L3 | Government ID + liveness | Creator onboarding, payout access, adult calling minor |

### Third-party identity verification providers

Recommended providers (as of 2025):
- **Stripe Identity** — good integration with Stripe payments; document + selfie check
- **Onfido** — comprehensive ID + biometric; strong UK/EU coverage
- **Jumio** — enterprise-grade; strong for financial compliance
- **Veriff** — good developer experience; strong European presence
- **Yoti** — specialist in age assurance and UK Children's Code compliance

**Key requirements when using a provider:**
- Sign a Data Processing Agreement (DPA)
- Ensure they're certified (ISO 27001, SOC 2)
- Store only the verification outcome + timestamp, not the raw ID document
- Define clear data retention limits for verification data

## 7. Account Recovery

### Email-based recovery

1. User requests password reset → generate a secure token (128 bits minimum entropy, single-use, 1-hour expiry)
2. Send link containing token to registered email
3. User clicks link → token validated server-side, user authenticated for reset only
4. After successful reset: invalidate token, invalidate all existing sessions, log the event

### Recovery codes (for MFA loss)

- Generate 10 single-use codes (8+ characters each, random) at MFA setup time
- Show codes once; user must save them
- Each code is hashed and stored (like a password)
- On use: hash the input, find matching stored hash, mark as used
- After all codes used: prompt user to regenerate

### Account recovery for admins

- Require identity verification (not just email) for admin account recovery
- Log all admin account recovery events
- Alert security team on admin account recovery attempts
